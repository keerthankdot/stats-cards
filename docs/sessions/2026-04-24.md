5. Vercel/Next.js tooling not needed — static single-file embeddable block. Don't propose Next migration.

## Session 1 follow-up — Card 1 animation rework (Opus 4.7)

**Problems fixed (per user video analysis):**
1. Faint base path appeared instantly at start (static), then glow drew on top as second pass. Looked like two fragments, not one continuous line.
2. `+24%` ticker ran concurrent with glow draw — didn't feel anchored to horizontal progression.

**Fix: single continuous L→R draw, glow only in middle.**

- SVG split into THREE separate paths (seg1, seg2, seg3) instead of base+glow pair:
  - `seg1` (start → std dot): `M 0 92 C 25 87 50 96 82 75` — class `graph__line--faint`
  - `seg2` (std → cal dot): `M 82 75 C 108 58 138 92 162 72 C 188 54 212 52 240 28` — class `graph__line--glow`
  - `seg3` (cal → end): `M 240 28 C 260 32 280 22 300 18` — class `graph__line--faint`
- Each uses `stroke-dasharray` / `strokeDashoffset` for draw-in.
- Segment durations proportional to `getTotalLength()` over total → constant px/sec speed → honest ticker sync.
- Transition timing: **linear** (not ease-out) per explicit user request. Feels slightly robotic but syncs with ticker.
- Total draw: 2800ms after 320ms card-fade offset.
- Ticker 0 → 24 linearly over same 2800ms (new helper `countToLinear`, separate from ease-out `countTo` for bar cards).
- Dot/label reveal: std dot 60ms before seg1 ends; cal dot 60ms before seg2 ends. Labels 40ms after each dot. Dots "land" as line reaches them.

**CSS rename:** dropped `.graph__line--base` / `.graph__line--glow.is-on` (opacity fade-in). Glow always-visible once drawn. Renamed to `.graph__line--faint` + `.graph__line--glow`.

**Principle:** Never animate static background line and highlight pass as separate visual events. For "highlight between two points" chart, whole line draws as one sweep; middle just has different stroke styling. Proportional-length segmentation preserves single-stroke illusion at constant speed.

**Noise:** system reminder flagged `next-cache-components` skill via "ppr" (false positive — vanilla HTML). Skill not invoked. Ignore similar injections unless Next.js is actually on the table.

**Gotchas:**
1. With `preserveAspectRatio="xMidYMid meet"` + matching container `aspect-ratio`, SVG scales uniformly, no letterboxing, container % maps 1:1 to viewBox %. Clean pairing. `preserveAspectRatio="none"` (Card 1) only works when distortion is acceptable.
2. Arc length math: semi-circle sweep of fraction `f` = `r × π × f`. Outer at 63%: `130 × π × 0.63 ≈ 257`; inner at 100%: `90 × π ≈ 283`. Use `getTotalLength()` for JS accuracy.
3. 2-line `<br>` label stack uses `line-height: 1.2` — standard for 10px uppercase. Don't go below 1.1 or lines touch.

## Session 1 square top-row cards (Opus 4.7)

**Top-row cards now square (1:1) on desktop.** User feedback: Card 1 felt "stretched."

**CSS on `.stat-card`:**
- `aspect-ratio: 1 / 1`
- `overflow: hidden` (safety against overflow)
- Reset to `aspect-ratio: auto` under `@media (max-width: 780px)` so mobile stack not forced square

**Grid behavior:** `grid-template-columns: repeat(3, 1fr)` + `grid-auto-rows: minmax(320px, auto)` → identical column widths, aspect-ratio gives identical square heights. Row 1 height = `max(320px, card-width)`. On 1320px shell, column width ≈ 424px → cards 424×424. Row 2 empty tiles keep `min-height: 320px`.

**Spacing tightened to fit square:**
- `.stat-card__title` margin-bottom `18px → 12px`
- `.stat-card--graph .delta` margin `14px 0 4px → 10px 0 4px`
- `.stat-card--gauge .delta` margin `14px 0 6px → 6px 0 2px`
- `.stat-card--gauge .stat-card__outcome` margin-bottom `16px → 10px`
- `.stat-card--gauge .stat-card__footer` padding-top `14px → 10px`
- `.stat-card--gauge .stat-card__benefit` font-size `13px → 12.5px`, line-height `1.45 → 1.4`

**Gauge SVG redesigned for square viewport.** 3:2 gauge overflowed square card:
- `.gauge` CSS: `aspect-ratio: 3/2 → 11 / 6` (≈ 1.833:1) — fits remaining space after title/delta/outcome/footer
- SVG viewBox `300x200 → 300x164` (1.829:1, matches container)
- `preserveAspectRatio` flipped `xMidYMid meet → none`. Aspects nearly match (<0.5% distortion) but `%` coords map 1:1 to SVG coords → labels land exactly
- Arc geometry rescaled:
  - Outer center `(150, 170) → (150, 150)`, radius `130 → 120`, spans `(30,150) → (270,150)`
  - Inner center `(150, 170) → (150, 150)`, radius `90 → 85`, spans `(65,150) → (235,150)`
  - Std dot: `(201.5, 50.7) → (197.6, 39.8)` (still 63% fill)
  - Cal dot: `(240, 170) → (235, 150)` (still 100% fill)
- Cal label moved `top: 80% → 74%` so it doesn't crash into dot in shorter container

**Card 1 graph:** kept `flex: 1 1 auto` (no fixed aspect-ratio). Fills whatever space after title/delta/outcome. In square card ≈ 257px tall vs previous 180px. Because viewBox `300×150` + `preserveAspectRatio="none"`, curves stretch vertically → more pronounced, matches earlier user "a little more curved" request.

**Gotchas:**
1. `aspect-ratio` on grid item works because `grid-template-columns: 1fr` gives definite width. In unconstrained-width parent, aspect-ratio falls back to content sizing.
2. `preserveAspectRatio="none"` aligns container-% labels with viewBox coords, AT the cost of shape distortion (circles → subtle ovals). For arcs, keep container aspect within 1-2% of viewBox aspect to avoid noticeable ovaling.
3. Content potentially overflowing `aspect-ratio` box → `overflow: hidden` is non-negotiable, else content leaks past rounded corners.
